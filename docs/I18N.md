# i18n — الترجمة والدعم متعدد اللغات

> **هذا الملف هو المرجع المعتمد لأي عمل ترجمة في المنصّة.**
> قبل أن تكتب نصّاً جديداً في صفحة Razor — اقرأ هذا الملف.

## Why this exists (لماذا)

المنصّة ثنائيّة اللغة (عربي/إنجليزي) من اليوم الأوّل. كل نصّ ظاهر للمستخدم
في Blazor يجب أن يمرّ عبر طبقة ترجمة موحَّدة — لا نصوص مكتوبة مباشرةً،
ولا شرطيّات `Store.Ui.IsArabic ? "ع" : "en"` منتشرة في الصفحات.

السبب: اللغة حالة (state) وليست ثابتاً. المستخدم قد يبدّلها في أي لحظة،
والصفحة يجب أن تعيد الرسم (re-render) مع النصوص الجديدة — هذا لا يعمل
إن كانت النصوص مجمَّدة في HTML.

## The approved pattern (الطريقة المعتمدة)

طبقتين:

1. **مكتبة مشتركة** — `libs/frontend/ACommerce.L10n.Blazor/`
   - تُعرّف العقود: `ITranslationProvider`, `ILanguageContext`
   - تُوفّر القاعدة: `EmbeddedTranslationProvider` (صنف مجرّد)
   - تُوفّر واجهة Razor: `L` — فيه `L["key"]` و `L.T(ar, en)`
   - تُوفّر تسجيل DI: `AddEmbeddedL10n<TProvider, TLangCtx>()`

2. **لكل تطبيق** — `Apps/{App}/Frontend/{App}.Web/Store/`
   - `AppLangContext.cs` — جسر قراءة اللغة من `AppStore.Ui.Language`
   - `L.cs` — صنف `{App}Translations` يرث `EmbeddedTranslationProvider`
     ويحمل القاموسين `_ar` و `_en` كحقول `static readonly`

المرجع الرسميّ — انسخ منه عند إنشاء تطبيق جديد:
- `Apps/Order.V2/Customer/Frontend/Order.V2.Web/Store/L.cs`
- `Apps/Order.V2/Customer/Frontend/Order.V2.Web/Store/AppLangContext.cs`

## Step-by-step — إضافة الترجمة لتطبيق جديد

### 1. أضف المرجع للمكتبة في `{App}.Web.csproj`

```xml
<ProjectReference Include="..\..\..\..\..\libs\frontend\ACommerce.L10n.Blazor\ACommerce.L10n.Blazor.csproj" />
```

### 2. أنشئ `Store/AppLangContext.cs`

```csharp
using ACommerce.L10n.Blazor;

namespace {App}.Web.Store;

public sealed class AppLangContext : ILanguageContext
{
    private readonly AppStore _store;
    public AppLangContext(AppStore store) { _store = store; }
    public string Language => _store.Ui.Language;
    public bool   IsRtl    => _store.Ui.Language != "en";
}
```

### 3. أنشئ `Store/L.cs` بقاموسين

```csharp
using ACommerce.L10n.Blazor;

namespace {App}.Web.Store;

public sealed class {App}Translations : EmbeddedTranslationProvider
{
    protected override IReadOnlyDictionary<string, string> Ar => _ar;
    protected override IReadOnlyDictionary<string, string> En => _en;

    private static readonly Dictionary<string, string> _ar = new()
    {
        ["app.name"]       = "اسم التطبيق",
        ["common.loading"] = "جارٍ التحميل…",
        // …
    };

    private static readonly Dictionary<string, string> _en = new()
    {
        ["app.name"]       = "App Name",
        ["common.loading"] = "Loading…",
        // …
    };
}
```

> **ملاحظة**: لا تُعرِّف صنف `L` محليّاً. استخدم `ACommerce.L10n.Blazor.L`
> الذي تسجّله `AddEmbeddedL10n` تلقائيّاً. (بعض التطبيقات القديمة فيها
> `L` محليّ ناتج عن تكرار تاريخيّ — لا تقلّده في تطبيق جديد.)

### 4. سجّل في `Program.cs`

```csharp
using ACommerce.L10n.Blazor;

builder.Services.AddEmbeddedL10n<{App}Translations, AppLangContext>();
```

هذا السطر يسجّل ثلاثة خدمات Scoped:
- `ILanguageContext` → `AppLangContext`
- `ITranslationProvider` → `{App}Translations`
- `L` → الواجهة الجاهزة للحقن في Razor

### 5. أضف `@using` في `Components/_Imports.razor`

```razor
@using ACommerce.L10n.Blazor
```

### 6. استخدمها في الصفحات

```razor
@page "/home"
@inject L L

<PageTitle>@(L["app.name"]) — @(L["home.title"])</PageTitle>

<h1>@(L["home.title"])</h1>
<p>@(L["home.subtitle"])</p>

<AcButton Text="@(L["common.submit"])" OnClick="Save" />

@if (L.IsRtl)
{
    @* منطق خاصّ بـ RTL إن لزم *@
}
```

## The three write-modes — متى تستخدم كلّ واحدة

| أسلوب | متى | مثال |
|---|---|---|
| `L["key"]` | **الافتراضيّ لكل نصّ جديد**. كل نصّ ثابت ينتظر المستخدم لرؤيته | `<h1>@(L["home.title"])</h1>` |
| `L.T(ar, en)` | استثناء مؤقّت أثناء الهجرة. نصّ واحد نادر الاستخدام، لا تبغى تسجيله في القاموس | `@L.T("غير معروف", "Unknown")` |
| `Store.Ui.IsArabic ? ar : en` | **استعمال مشروع محدود** — انظر القائمة أدناه | مذكور في حالات استثنائيّة |

### `L.T(ar, en)` — خطوة انتقاليّة، ليست وجهة

`L.T("نص", "text")` مقبول مؤقّتاً أثناء تحويل صفحة قديمة. لكن لا تتركه
دائماً. القاعدة: **كل `L.T()` يُستبدل بـ `L["key"]` قبل قبول الـ PR**.
الاستثناء الوحيد: نصّ تشخيصيّ (diagnostic) يُعرض مرّة واحدة ولا قيمة
لتكراره في القاموس.

### متى يُسمح بـ `Store.Ui.IsArabic`؟

ثلاث حالات فقط:

1. **زر تبديل اللغة** — بطبيعته يعرض اللغة المعاكسة:
   ```razor
   <AcButton Text="@(Store.Ui.IsArabic ? "English" : "العربية")" />
   ```

2. **بيانات API فيها `NameAr` و `NameEn`**:
   ```csharp
   Label = Store.Ui.IsArabic ? category.NameAr : category.NameEn
   ```
   (الترجمة في الخلفيّة لا في القاموس.)

3. **تمرير `bool` لمكوّن شِدّة (widget)** يحتاج اتجاه/إعداد داخليّ:
   ```razor
   <AcAuthGuard IsArabic="@Store.Ui.IsArabic" … />
   ```

غير هذه الحالات — `Store.Ui.IsArabic` في صفحة Razor **علامة تحذير** (code
smell). حوّله لمفتاح في القاموس.

## Naming convention — تسمية المفاتيح

- kebab مفصول بنقاط: `domain.feature.label`
- مجموعات قياسيّة موجودة في كل تطبيق:
  - `app.*` — اسم التطبيق وشعاره
  - `common.*` — أزرار وإشعارات وعناصر مشتركة (loading, save, cancel…)
  - `nav.*` — عناصر الشريط السفليّ/العلويّ
  - `auth.*` — تسجيل الدخول والـ OTP
  - `settings.*` — الإعدادات
- مجموعات خاصّة بالصفحة: `home.*`, `cart.*`, `checkout.*`, `profile.*`…

أمثلة من `Order.V2.Web/Store/L.cs` (المرجع):
```
app.name, app.tagline
common.save, common.cancel, common.loading, common.error
nav.home, nav.cart, nav.profile
auth.login.title, auth.phone, auth.otp
home.title, home.subtitle
cart.empty_title, cart.checkout
```

## Rules (القوانين)

1. **لا نصّ ثابت ظاهر بدون مفتاح.** كل `<h1>`, `<p>`, `Title="…"`, placeholder
   يجب أن يمرّ عبر `L[...]`. إن كان النصّ يظهر للمستخدم — فهو قابل للترجمة.

2. **تناسق المفاتيح بين القاموسين.** إن أضفت `_ar["x.y"]` يجب أن يكون
   `_en["x.y"]` موجوداً. قيمة فارغة = خطأ.

3. **لا ترميز Unicode في الكود خارج القاموس.** يعني: لا تكتب نصوصاً
   عربيّة في `.razor` أو `.cs` إلّا داخل `_ar` في `L.cs` (أو كسلسلة مرّة
   استثنائيّة حسب قائمة `Store.Ui.IsArabic` أعلاه).

4. **تسجيل DI فقط في `Program.cs` الواحد.** لا تُسجِّل `ITranslationProvider`
   مرّتين ولا تخلط `AddSingleton` مع `AddScoped` — استخدم
   `AddEmbeddedL10n<T, C>()` الموحَّدة.

5. **المفتاح الناقص يُرجع نفسه.** إذا ظهر `home.title` في الواجهة بدل
   نصّ حقيقيّ — مفتاح مفقود من القاموس. أضفه فوراً في اللغتين.

6. **احفظ بيانات API كما هي.** إن كان الـ endpoint يرجع `NameAr`/`NameEn`
   فلا تُنشئ مفاتيح ترجمة لها — استخدم `Store.Ui.IsArabic` لاختيار الحقل
   المناسب (الحالة الثانية من استثناءات `Store.Ui.IsArabic`).

## How to extend — إضافة مزوِّد ترجمة جديد

الفكرة المعماريّة: `ITranslationProvider` عقد قابل للتبديل. حاليّاً
`EmbeddedTranslationProvider` يقرأ من قاموس ثابت، لكن يمكن استبداله بـ:

- `ApiTranslationProvider` — يجلب من `/api/translations` ويُخزّن
- `ResxTranslationProvider` — يستخدم `.resx` الكلاسيكيّ لـ .NET
- `DbTranslationProvider` — CMS داخليّ يعدّله محرّر محتوى بدون نشر

الانتقال = تغيير سطر واحد في `Program.cs`. الصفحات لا تتغيّر لأنّها
تعتمد على الواجهة `L` فقط.

## Verification checklist — تحقّق قبل قبول الـ PR

- [ ] لا توجد `Store.Ui.IsArabic` جديدة خارج الحالات الثلاث المسموحة
- [ ] كل `L.T("…", "…")` مستخرج لمفتاح في `_ar` و `_en`
- [ ] عدد مفاتيح `_ar` = عدد مفاتيح `_en`
- [ ] لا قيم فارغة `""` في أيّ قاموس
- [ ] تبديل اللغة في التطبيق يعمل ويعيد رسم كل الصفحات
- [ ] الصفحات بالعربيّة تظهر بـ RTL صحيح
- [ ] النصوص العربيّة تستخدم محارف قياسيّة (U+0600–U+06FF)، لا محارف مشابهة
